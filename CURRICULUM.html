              Next.js 16 + Hono + Prisma + Auth.js v5

              A migration curriculum for a senior C# / .NET + functional React/TS developer. Companion to ROADMAP.md.

              How this works

              Every exercise triggers the failure mode first. If you complete an exercise without seeing the red error before the green tick, you skipped a step.

              Time budgets account for setup. Nine exercises are marked ⭐; these are the highest-value ones to prioritise.

              Total budget: ~39 hours of focused work, plus 5–7 hours of one-time environment setup (GitHub OAuth registration, Cloudflare account setup, font installation, and editor setup). At 2 hrs/day, four weeks. At 4 hrs/day, ten working days.

              Critical 2026 notes (read first)

              Three facts that are easy to get wrong; propagating any wastes hours. Verify each first.

              The Hono JWK middleware CVE is CVE-2026-22818 (advisory GHSA-3vhc-576x-3qv4), fixed in Hono 4.11.4+. Note that CVE-2026-27804 is a different package entirely (a Parse Server vulnerability). When auditing your Hono version, search the GHSA ID, not that CVE.
              @auth/core/jwt decode() takes { token, secret, salt }, not { token, secret }. The salt parameter is required and used together with the secret to derive the decryption key. Omitting it produces an unhelpful decryption error. See the authjs.dev/reference/core/jwt reference.
              revalidateTag(tag) single-arg is deprecated with a TS warning, not a hard compile error. It still works in Next.js 16.2.6. Use the two-arg form revalidateTag(tag, profile) in all new code; don't expect the compiler to force you to fix legacy calls.

              Prerequisites: set these up once

              Several mid-curriculum exercises assume infrastructure is already in place. Set this up in one batch up front. Allocate ~4 hours end-to-end on a fresh Windows 11 box.

              Choose your orchestration path

              The "develop on Windows-native PowerShell" pain is mostly an orchestration problem: running Postgres + PgBouncer + Redis + Hono + Next.js together with health-check ordering, service discovery, and unified logs. Two paths solve it differently. Both lead to the same exercises.

              Path A: Aspire AppHost orchestrating npm apps + containers

              If you have .NET on this machine and already know Aspire, use this path. Aspire's AddNpmApp + WithReference + WaitFor replaces docker-compose.yml + process-compose + manual env-var wiring with one C# file you can debug. The dashboard at https://localhost:18888 gives unified logs, traces, and metrics across every resource, with more depth than process-compose's TUI.

              What Aspire does NOT solve:
              • pnpm install on NTFS is ~2× slower than ext4 (still better than the 5× /mnt/c crossover penalty)
              • Enable Windows Developer Mode (Settings → Privacy & Security → For Developers) or pnpm's symlinks fail with EPERM: A required privilege is not held by the client
              • CRLF/LF is not solved by .gitattributes alone. See the dedicated section below: the Git for Windows installer defaults core.autocrlf=true, which silently fights the attribute file
              • Next.js HMR on NTFS has edge cases on very large component trees
              • Prisma binaryTargets still needs Windows + Linux entries (same as WSL2 path)
              CRLF/LF setup (.gitattributes alone is insufficient)

              The most common Windows-native failure mode is Husky pre-commit hooks dying with /usr/bin/env: 'bash\r': No such file or directory. The hook file was committed with CRLF endings before .gitattributes normalized them, and bash chokes on the \r in the shebang.

              .gitattributes is necessary but not sufficient. The full checklist:

              1. Repo root .gitattributes: 
                # Default: text files normalize to LF in repo and on checkout
* text=auto eol=lf

# Explicit binary list — prevents Git from mangling them
*.png   binary
*.jpg   binary
*.ico   binary
*.pdf   binary
*.woff  binary
*.woff2 binary

# Shell scripts and hooks MUST be LF regardless of platform
*.sh    text eol=lf
.husky/* text eol=lf
              2. Repo root .editorconfig: 
                root = true
[*]
end_of_line = lf
insert_final_newline = true
charset = utf-8
indent_style = space
indent_size = 2
              3. Every dev runs this once (Git for Windows defaults autocrlf=true, which overrides this): 
                git config --global core.autocrlf false
git config --global core.eol lf
              4. One-time renormalization commit after .gitattributes is committed: 
                git add --renormalize .
git commit -m "chore: normalize line endings"
                Without this, existing files keep their original endings; the attributes only affect files Git touches afterward.
              5. Install Husky after steps 1–4 are merged. Then verify each hook file: 
                file .husky/pre-commit
# Want: ".husky/pre-commit: ASCII text"
# Bad:  ".husky/pre-commit: ASCII text, with CRLF line terminators"
              6. Workspace .vscode/settings.json: 
                {
  "files.eol": "\n"
}
              7. CI safety net. Add a job step that fails the build on stray CRLF in tracked text files: 
                ! git ls-files --eol | grep -E 'i/crlf'
                Without this, a teammate who joins later without running step 3 silently reintroduces CRLF, and you find out via confusing hook failures weeks later.

              All seven steps are required. Skipping step 3 or step 7 reintroduces CRLF the moment a new teammate joins or a Bash-shebanged tool is added to scripts/.

              The AppHost.cs

              Create an orchestrator/ folder at your monorepo root with a minimal Aspire AppHost project. The following covers the whole stack:

              // orchestrator/AppHost.cs
var builder = DistributedApplication.CreateBuilder(args);

// Postgres 17 — exposed on host port 5433 (avoids collision with any system pg)
var postgres = builder.AddPostgres("postgres", port: 5433)
    .WithDataVolume()       // named volume, persists across restarts
    .WithPgAdmin();         // optional: spins up pgAdmin alongside, served via dashboard

var appdb = postgres.AddDatabase("appdb");

// PgBouncer in transaction mode — sets up the Phase 2 footgun exercise
var pgbouncer = builder.AddContainer("pgbouncer", "edoburu/pgbouncer")
    .WithEnvironment("DB_HOST", postgres.Resource.Name)
    .WithEnvironment("DB_USER", "postgres")
    .WithEnvironment("DB_PASSWORD", postgres.Resource.PasswordParameter)
    .WithEnvironment("DB_NAME", "appdb")
    .WithEnvironment("POOL_MODE", "transaction")
    .WithEnvironment("AUTH_TYPE", "scram-sha-256")
    .WithEndpoint(targetPort: 5432, port: 6432, name: "pgbouncer")
    .WaitFor(postgres);

// Redis — used in the Phase 5 deep-dive (JWT revocation blocklist)
var redis = builder.AddRedis("redis", port: 6379)
    .WithDataVolume();

// Hono backend (Node) — Aspire treats it as an npm app
var api = builder.AddNpmApp("api", "../services/api", "dev")
    .WithReference(appdb)              // injects ConnectionStrings__appdb
    .WithReference(redis)              // injects ConnectionStrings__redis
    .WithHttpEndpoint(port: 3001, env: "PORT")
    .WaitFor(postgres)
    .WaitFor(redis)
    .PublishAsDockerFile();            // for prod deploys

// Next.js frontend — also npm app
var web = builder.AddNpmApp("web", "../apps/web", "dev")
    .WithReference(api)                // service discovery: api URL injected
    .WithReference(appdb)              // direct Prisma access from RSCs
    .WithHttpEndpoint(port: 3000, env: "PORT")
    .WaitFor(api)
    .PublishAsDockerFile();

builder.Build().Run();

              Path B: WSL2 + process-compose

              Take this path if you don't have .NET on this machine, if a Mac/Linux teammate joins the project, or if you want Linux dev parity with the production container environment. Tools are all native and file performance is optimal.

              Performance footgun: Never store project files under /mnt/c/.... The WSL2 9P bridge to NTFS is 3–5× slower for pnpm install and prisma generate. Keep everything under ~/projects/ inside WSL2's native ext4. Access from Windows via \\wsl$\Ubuntu\home\<user>\projects.

              Shell stack (inside WSL2)

              Shell stack: Zsh + Oh My Zsh + Starship. Avoid Fish; its non-POSIX syntax breaks copy-pasted shell snippets. Use WezTerm on Windows as the terminal emulator (GPU-accelerated, with a Lua config consistent with Neovim).

              Two-stage install: Run Stage 1 in your default bash WSL2 session. Then close and reopen your terminal so zsh loads with fnm in PATH. Then run Stage 2 in the new zsh session. Sourcing .zshrc from inside a bash script does not properly inject fnm into the current shell.

              Docker stack: Postgres + PgBouncer + Redis

              Use Rancher Desktop (free, Apache 2.0) over Docker Desktop; Docker Desktop requires a paid license above the 250-employee / $10M-revenue threshold. Both use WSL2 as the VM backend. In Rancher Desktop → Preferences → WSL Integration → enable Ubuntu-24.04, then docker --version works inside WSL2.

              Drop this docker-compose.yml in dev/ and you have your Phase 2 and Phase 6 infrastructure ready. The PgBouncer POOL_MODE: transaction is intentional: it sets up the Phase 2 footgun exercise.

              # dev/docker-compose.yml — usage: docker compose -f dev/docker-compose.yml up -d
services:
  postgres:
    image: postgres:17-alpine
    restart: unless-stopped
    environment:
      POSTGRES_USER: ${POSTGRES_USER:-dev}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:-devpassword}
      POSTGRES_DB: ${POSTGRES_DB:-appdb}
    volumes: [pgdata:/var/lib/postgresql/data]
    ports: ["5433:5432"]
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER:-dev} -d ${POSTGRES_DB:-appdb}"]
      interval: 10s
      timeout: 5s
      retries: 5

  pgbouncer:
    # bitnami/pgbouncer moved behind a paid subscription in 2024-2025 — using the official
    # pgbouncer org image instead. Note: env var names differ from the old bitnami scheme.
    image: edoburu/pgbouncer:latest
    restart: unless-stopped
    environment:
      DB_HOST: postgres
      DB_PORT: 5432
      DB_USER: ${POSTGRES_USER:-dev}
      DB_PASSWORD: ${POSTGRES_PASSWORD:-devpassword}
      DB_NAME: ${POSTGRES_DB:-appdb}
      POOL_MODE: transaction              # the footgun: Prisma interactive tx breaks here
      MAX_CLIENT_CONN: 100
      DEFAULT_POOL_SIZE: 10
      AUTH_TYPE: scram-sha-256
    ports: ["6432:5432"]                  # exposed as 6432 on host, container's pgbouncer listens on 5432
    depends_on:
      postgres:
        condition: service_healthy

  redis:
    image: redis:7-alpine
    restart: unless-stopped
    command: redis-server --appendonly yes --requirepass ${REDIS_PASSWORD:-devredis}
    volumes: [redisdata:/data]
    ports: ["6379:6379"]
    healthcheck:
      test: ["CMD", "redis-cli", "-a", "${REDIS_PASSWORD:-devredis}", "ping"]
      interval: 10s

volumes:
  pgdata: {}
  redisdata: {}

              Exercise: two-stage, end-to-end then rotate

              Deep dive: JWT revocation via Redis JTI blocklist

              90 min DEPTH

              Stateless JWTs cannot be revoked without infrastructure.               OneUptime — Token Blacklist for JWT Revocation with Redis (March 2026)
              Deep-dive exercise: JTI blocklist + user-level revocation + p99 benchmark
              Add jti everywhere

              jti = crypto.randomUUID() in the jwt callback. Hono verification middleware reads it.

              Redis JTI blocklist

              Add a Redis client (Upstash or local; the prereqs docker-compose has Redis ready). In Hono middleware after signature verification, call redis.get(jti) and 401 if it exists.
